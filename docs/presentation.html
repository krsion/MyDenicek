What is Denicek?

A computational substrate for document-oriented end-user programming — Petříček, 2022
  • Documents are tagged trees — records, lists, primitives, references
  • Addressed by selector paths: /items/2/name, /speakers/*

Four key experiences:

  • Programming by demonstration — record & replay edit sequences
  • Schema evolution — structural edits (wrap, rename, update tags)
  • Collaborative editing — but uses error-prone OT with central server
  • Formula recomputation — computed values from references

Project Goal

Make Denicek local-first and implement formative examples
that demonstrate its end-user programming capabilities.

What is Local-First?

  • Multiple peers edit the same document concurrently
  • Works offline — sync when reconnected
  • All peers converge to the same state (strong eventual consistency)
  • No central authority — the server is just a relay
  • Data lives on your device, not in the cloud

Building From Scratch

The same conference list can be built from an empty document.

Start: { "$tag": "root" } — command bar at the bottom

Step 1: Add Structure

1 Add a heading to root:

/ add heading {"$tag":"h2","text":"Conference Speakers"}

2 Add an empty speakers list:

/ add speakers {"$tag":"ul","$items":[]}

The / selector targets the root. JSON values create subtrees.

Step 2: Add Speakers

Push items with "Name, email" format:

/speakers pushBack {"$tag":"li",
  "contact":"Ada Lovelace, ada@example.com"}
/speakers pushBack {"$tag":"li",
  "contact":"Grace Hopper, grace@example.com"}
/speakers pushBack {"$tag":"li",
  "contact":"Alan Turing, alan@example.com"}

Each pushBack is one event in the DAG. The event graph grows.

Step 3: Refactor List → Table

Structural edits transform the document in place:

/speakers updateTag table    ← ul → table
/speakers/* updateTag tr     ← li → tr (wildcard!)

The * wildcard applies the edit to all children at once.

These are recorded events — a second peer receives them via sync and sees the same table.

Journey: Automerge

✅ Pros

  • Mature CRDT library
  • Rich text, lists, maps out of the box
  • Large community

❌ Why I moved on

  • No move operation — moving a node = delete + insert → loses identity
  • Structural edits (wrap, rename) don't map naturally
  • Opaque IDs, not Denicek's path selectors

Journey: Loro

✅ Pros

  • Has a movable tree CRDT
  • Built-in undo manager
  • Event subscriptions for recording

❌ Why I moved on

  • Still uses opaque node IDs, not paths
  • Custom structural edits (wrap, rename fields, update tags) — not natively supported
  • Replay with retargeting requires selector OT — Loro doesn't provide it
  • We actually need paths.

Solution: Custom OT-based Event DAG

The Denicek editing model is inherently OT-shaped.

  • Event DAG with vector clocks for causal ordering
  • Deterministic topological replay — all peers replay in same order
  • Selector-based OT — structural edits transform paths of concurrent edits
  • Rename, wrap, unwrap, delete, copy — each has hand-written OT rules
  • Zero external dependencies — pure TypeScript

Key Insight: Why Paths Matter

ID-based (Automerge/Loro)

  • node_abc123.set("name", "Ada")
  • Need lookup table to find nodes
  • Wildcards? Not native.
  • Replay targets a fixed node

Path-based (mydenicek)

  • dk.set("speakers/0/name", "Ada")
  • Human-readable, inspectable
  • speakers/* → all items
  • Replay + OT follows renames & wraps

How Convergence Works

  1. Each peer records edits as events in a causal DAG
  2. Events carry vector clocks — tracks happens-before
  3. To materialize: topological sort (Kahn's algo + deterministic tie-breaking)
  4. Concurrent structural edits → OT transforms selectors
  5. Same events → same sort → same transforms → same document

Validated by property-based tests (fast-check) and a random fuzzer.

Custom edits only need to be registered on the client.

Testing & Validation

  • 205+ unit tests — core operations, OT rules, edge cases
  • Property-based tests (fast-check) — random concurrent edit sequences, verify convergence
  • Random fuzzer — long-running stress test with arbitrary operations
  • 11 sync E2E tests — basic sync, late join, concurrent edits, reconnect, pause/resume
  • Playwright E2E — browser tests for the full UI flow
  • CI/CD — GitHub Actions: lint, check, test, build, deploy on every push

What I Learned

  • CRDTs aren't always the right abstraction — when your model is path-based, OT is more natural
  • Compound operations can't be atomic in local-first — wrap = create + move → design for partial application
  • Deterministic replay is the core of convergence — get the sort right, get OT right, everything follows
  • The sync server should be dumb — pure relay, no materialization, no edit knowledge
  • Property testing catches bugs that hand-written tests miss

Future Work

  • Optimized implementation — the current replay-from-scratch engine is correct but slow for large documents. Build an incremental version that caches partial materializations.
  • Model-guided fuzzing — use this implementation as a reference oracle to fuzz-test the optimized one. Generate random edit sequences, run both, compare outputs.
  • myDatnicek — a data-oriented variant built on the optimized core, capable of handling larger datasets (spreadsheets, databases) while preserving local-first collaboration.
  • Compaction — garbage-collect old events once all peers have seen them, reducing DAG size.
