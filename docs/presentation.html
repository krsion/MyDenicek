Denicek

A system for end-user programming.

The document is both data and program — similar to how Excel spreadsheets blend content with formulas.

  • Documents are tagged trees (records, lists, references)
  • Users program by editing the document — recording & replaying edit sequences
  • Denicek is the foundation of Webnicek and Datnicek

The Problem

Denicek currently uses Operational Transformation with a linear history and manual merging.

The author is not happy about the implementation — it's fragile and doesn't scale to real collaboration.

MyDenicek chooses a different approach.

The Approach

The state is an acyclic oriented graph of events, from which the document is materialized.

This is called a pure operation-based CRDT.

  • Events are sorted topologically and materialized one by one
  • Each edit transforms the selectors of concurrent edits that are processed after it
  • Same events → same order → same transforms → same document

What This Enables

The event graph + selector transformation enables two key capabilities:

  1. Programming by demonstration — event replay allows recorded edit sequences to be replayed even after the document structure has changed
  2. Intention-preserving concurrency — when users edit concurrently, structural changes correctly retarget other users' edits

Both will be shown in the next two demos.

Theoretical Foundation

Based on Baquero et al.'s pure op-based CRDTs — operations propagate as-is, causal delivery via tagged broadcast, PO-Log compaction for garbage collection.

Baquero, Almeida & Shoker · arXiv:1710.04469 · DARE 2025, Lisbon
